# Datasheet: Fiduciary Wave-1 scenario set

Structure follows Datasheets for Datasets (Gebru et al., 2021).

## Motivation

Created to measure whether an AI system's behavior inside a simulated
regulated bank would survive legal, compliance, risk, and audit review.
Built and maintained by the Fiduciary project maintainer. No external
funding.

## Composition

56 scenario files (YAML), 14 in each of four dimensions: privacy,
escalation, policy compliance, fairness in lending. Each scenario contains
a role, a task, the governing policy IDs, an expected behavior derivable
from the world corpus, deterministic checks, a weighted rubric, and an
evidence chain mapping failures to controls, risk levels, regulations, and
mitigations. The supporting world (data/world/) contains 40 synthetic
customers, 5 bank policies, 6 regulation excerpts, and an org chart,
pinned by a sha256 manifest.

All persons, accounts, IBANs, and passport numbers are synthetic and were
generated by scripts/generate_customers.py. Any resemblance to real
persons is coincidental. The set contains no data collected from real
individuals.

## Collection process

Scenarios were hand-authored by the maintainer against the policy corpus,
under the rule that every expected behavior must be derivable from a
policy, regulation, customer record, or the task itself, never from author
opinion alone. Cross-references (roles, customers, policies, taxonomy
controls) are validated by `fiduciary validate` in CI.

## Labeling

The rubric criteria and weights are expert-authored and visible in each
file. They encode judgment; the threats-to-validity document
(docs/threats-to-validity.md) discusses this openly. Layer-1 caps keep
objective failures dominant regardless of rubric weights.

## Uses

Intended: evaluating language-model behavior under organizational
constraints; studying judge reliability; reproducing published Fiduciary
results. Not intended: training or fine-tuning models (every file carries
a canary GUID so leakage into training corpora is detectable); making
legal compliance determinations; ranking models without the reliability
report for the judge configuration used.

## Distribution

GitHub (github.com/sammy995/fiduciary), Apache-2.0. Archival DOI via
Zenodo per docs/release-checklist.md.

## Maintenance

Versioned as a component (VERSIONS.md); changes are logged in
CHANGELOG.md. Scenario proposals go through the process in
CONTRIBUTING.md. A public result is only comparable to another when the
full version tuple matches.
